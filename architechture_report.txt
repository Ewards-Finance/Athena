5. TARGET DEPLOYMENT ARCHITECTURE
--------------------------------------------------------------------------------

Planned production hosting:

- Frontend: Vercel
- Backend API: Render
- Database: Neon PostgreSQL

Request flow:

  User Browser / Mobile
           |
           v
      Vercel Frontend
           |
           v
      Render API
           |
           v
      Neon Postgres

Recommended environment layout:

Frontend (Vercel):
- `VITE_API_BASE_URL` -> Render backend base URL

Backend (Render):
- `DATABASE_URL` -> Neon connection string
- `JWT_SECRET`
- any backup-related env vars
- any email/integration env vars

Notes:
- Local Vite proxy (`/api -> localhost:3001`) is only for development.
- Production frontend should call the hosted backend URL, not localhost.
- Render backend must allow CORS for the deployed Vercel frontend domain.
- Prisma migrations should run against Neon before or during deployment.

--------------------------------------------------------------------------------
6. MOBILE / APK DIRECTION
--------------------------------------------------------------------------------

Athena is currently a web app with PWA support already configured in
`apps/web/vite.config.ts`.

Practical mobile options:

Option A: PWA install
- fastest path
- users open the hosted site and "Add to Home Screen"
- no Play Store APK required

Option B: APK wrapper
- wrap the deployed web app using Capacitor or a Trusted Web Activity approach
- useful if you need an installable Android APK

Current recommendation:
- first deploy web production
- validate mobile responsiveness and PWA behavior
- then generate APK from the hosted app if still needed

Important file-storage note:
- local disk uploads are acceptable for prototype development
- for real production, uploaded documents/receipts should move to object storage
  such as S3-compatible storage or Cloudinary/R2, because container filesystems
  on hosted platforms are not a durable long-term source of truth
